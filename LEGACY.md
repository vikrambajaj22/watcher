# Watcher Legacy Architecture (FAISS + TMDB Metadata Cache)

**Version**: Original Watcher (late 2025–early 2026)  
**Last commit using this approach**: 8b23e25 (mobile friendly ui)

This document describes the original architecture that Watcher used before the June 2026 refactor. This approach has been **deprecated and replaced** with the simpler LLM + TMDB discover flow documented in [README.md](README.md).

The legacy system was a sophisticated vector-search engine: it cached the entire TMDB metadata catalog (~1.3GB) in MongoDB, precomputed 384-dimensional embeddings for all movies/TV shows, and stored them in a FAISS index. Recommendations were generated by finding similar items in vector space (KNN), then applying LLM reasoning to rank and explain them. The app also offered "Will I Like?" predictions and an interactive t-SNE visualization of your watch history.

---

## Why We Changed

The original architecture had several pain points:

1. **Large metadata overhead** — storing ~1.3GB of TMDB metadata in MongoDB
2. **Sync complexity** — maintaining incremental vs. full TMDB sync logic
3. **Vector embedding costs** — constant embedding computation and model hosting
4. **FAISS index management** — rebuilds, versioning, cold-start latency
5. **Limited recommendation quality** — KNN + static LLM reasoning didn't adapt well to diverse tastes
6. **Data loss risk** — MongoDB on small VMs (e2-micro) suffered OOM crashes
7. **GCP costs** — keeping Cloud Run + GCE VM + GCS bucket alive cost

**The new approach** (LLM taste planner + TMDB discover) is simpler, cheaper, and produces better recommendations because:
- The taste planner dynamically understands the user's recent preferences
- Fresh TMDB candidates avoid stale metadata issues
- No large DB to maintain or sync
- Stateless — scale horizontally with ease

### Original Data Flow

1. **Initial TMDB Ingest** (one-time, then periodic):
   - Fetch TMDB daily export (gzipped newline-delimited JSON of all IDs)
   - Batch-fetch full metadata per ID (details, credits, keywords)
   - Store entire documents in MongoDB `tmdb_metadata` collection
   - Compute embeddings from metadata + credits
   - Build and persist FAISS index to disk + GCS bucket

2. **Incremental TMDB Sync** (periodic via sync_worker):
   - Use TMDB `/movie/changes` and `/tv/changes` endpoints
   - Fetch deltas, update MongoDB documents
   - Optionally recompute embeddings for changed items
   - Update FAISS index (in-place or rebuild)

3. **Watch History Sync** (periodic or on-demand):
   - Check Trakt `/sync/last_activities` for changes
   - If changed, fetch full `/sync/watched/{movie|tv}` and replace collection
   - Compute user vector from history (weighted embedding average)

4. **Recommendations**:
   - Build user vector from watch history embeddings
   - KNN search in FAISS index → return N nearest neighbors
   - Fetch metadata for candidates from MongoDB
   - LLM refiner: rank and add reasoning
   - Return top results

5. **Will I Like?**:
   - Fetch item metadata
   - Compute embedding for the item
   - Compare to user vector (similarity threshold 0.65)
   - LLM generates explanation

6. **Similar Items Search** (by TMDB ID or free-text):
   - User provides a movie/show ID or title
   - If title: resolve to TMDB ID and media type
   - Fetch embedding for the item from FAISS or compute on-the-fly
   - KNN search in FAISS index → return N most similar items
   - Fetch metadata for results from MongoDB
   - Optional filter: exclude already-watched items
   - Return ranked list with posters, genres, overview

7. **Visual Explorer**:
   - All watch history items
   - Apply t-SNE to embeddings → 2D coordinates
   - Cluster with K-means
   - Return interactive visualization

## FAISS Index Management (Legacy)

### Embedding Workflow

1. **Full Rebuild** (recommended for initial population):
   - Compute 384-dim embeddings for each metadata record using sentence-transformers
   - Build FAISS index (CPU, IDMap + IVF100 + Flat)
   - Persist sidecars: `labels.npy` (TMDB IDs), `vecs.npy` (vectors), `sidecar_meta.json` (metadata)
   - Upload to GCS bucket (e.g., `gs://watcher-faiss/v1/`)

2. **Incremental Upsert** (for single items):
   - Compute embedding for new/updated item
   - Attempt in-place FAISS add/remove
   - If not supported, schedule full rebuild

3. **Runtime Caching**:
   - Each Python process caches the FAISS index in memory after first load
   - Admin endpoints allow clearing per-process cache
   - External rebuilds require cache invalidation

### Example Initial Population

```bash
# Step 1: Sync TMDB metadata (no embeddings)
python - <<'PY'
from app.tmdb_sync import sync_tmdb
sync_tmdb('movie', full_sync=True, embed_updated=False)
sync_tmdb('tv', full_sync=True, embed_updated=False)
PY

# Step 2: Build embeddings and FAISS index
python -m app.faiss_rebuild_cli --dim 384 --factory "IDMap,IVF100,Flat"

# Step 3: Upload to GCS
gsutil -m cp tmdb.index labels.npy vecs.npy sidecar_meta.json gs://watcher-faiss/v1/
```

### Admin API Endpoints (Legacy)

- `GET /admin/faiss/status` — report index status, metadata, cached boolean
- `POST /admin/faiss/clear-cache` — clear in-process FAISS cache
- `POST /admin/faiss/rebuild` — spawn detached rebuild process
- `POST /admin/embed/full` — schedule full embedding + index build
- `POST /admin/embed/item` — upsert single item embedding
- `POST /admin/faiss/upsert-item` — low-level in-place upsert

## GCP Deployment (Legacy)

The legacy stack was deployed on GCP as:

- **Backend**: Cloud Run (8 GB memory, 2 CPU, 900s timeout) running FastAPI + sentence-transformers + FAISS
- **MongoDB**: Compute Engine e2-micro VM (1 GB RAM, persistent disk)
- **FAISS Index**: GCS bucket with volume mount or download-on-cold-start
- **Static UI**: Cloud Run (nginx) or Firebase Hosting
- **Networking**: Cloud NAT for VM outbound, firewall rules for Mongo port

**Cost**: ~$50–100/month when idle (backing storage, reserved IPs, minimal traffic)

### GCP Setup Steps

1. Create GCE VM (e2-micro), install MongoDB
2. Create MongoDB user, restore from dump
3. Build backend Docker image, deploy to Cloud Run with 8Gi memory
4. Build frontend, deploy to Cloud Run or static host
5. Create GCS bucket, upload FAISS index
6. Mount GCS bucket to backend via volume, or set download mode
7. Wire Trakt → backend callback URL, backend → Mongo internal IP

See original `DEPLOYMENT.md` for full steps (still in git history).

## Sync Worker (Legacy)

The `sync_worker.py` ran continuously (or via Cloud Scheduler) to:
- Check Trakt `/sync/last_activities` for changes
- If changed, fetch full watch history and update MongoDB
- Optionally trigger TMDB sync for new/changed items
- Periodically full-sync TMDB via export files and discover fallback

**Problem**: sync_worker was not deployed in production for most of the project, so syncs only happened on-demand (when user hit `/recommend` endpoint).

## Known Issues (Why We Changed)

### 1. Data Loss on MongoDB OOM

Small VMs (e2-micro, 1 GB RAM) would run out of memory during Trakt sync bursts (watch_history collection bulk writes), causing mongod to crash and corrupting the database. The `tmdb_metadata` collection (~1.3 GB) would vanish on recovery.

**Attempted mitigations**:
- Reduced WiredTiger cache size
- Added swap
- Moved to larger VM (e2-small)
- Finally: removed the need for persistent TMDB metadata entirely

### 2. TMDB Metadata Drift

Even with incremental sync, the metadata in MongoDB would drift from the authoritative TMDB API. New fields added to TMDB or changes to existing records would not be reflected without a full rebuild.

### 3. Embeddings Model Lock-In

Embeddings were computed with sentence-transformers 384-dim. Switching embedding models required full recomputation + FAISS rebuild.

### 4. Slow Cold Starts

Cloud Run cold start with FAISS index download (~2 GB) took 10–30 seconds. Mounting GCS bucket helped but added complexity.

### 5. LLM Reasoning was Static

The original KNN approach found similar items in vector space, but the LLM reasoning was applied post-hoc to rank them. This didn't adapt well when a user's taste evolved (e.g., "I want something completely different this time").

The new approach lets the taste planner re-analyze preferences for each request, making recommendations more adaptive.

### 6. Cost and Complexity

- GCP billing for Cloud Run + GCE VM + GCS + data transfer
- Schema migrations for MongoDB
- FAISS index versioning and rollback overhead
- Deployment scripts specific to GCP

Simpler single-VPS or Cloudflare Pages + VPS approach is cheaper and easier to understand.

## Migration Path (If You Decide to Return)

To revert to the legacy version:

1. Create a feature branch from the current main
2. Check out the old FAISS code from git history (commit `6139f65` or earlier)
3. Restore `app/embeddings.py`, `app/faiss_*.py`, `app/process/recommendation.py` (old version)
4. Restore `app/tmdb_sync.py` and sync logic
5. Update `app/api.py` to re-enable `/recommend/{media_type}` (old KNN path)
6. Restore MongoDB schema migrations if needed
7. Re-enable GCP deployment scripts from `DEPLOYMENT.md` (check git history)

The old code is preserved in git history; cherry-pick or rebase as needed.

## Summary of Changes

| Aspect | Old (FAISS) | New (LLM) |
|--------|------------|----------|
| TMDB metadata storage | Full cache in MongoDB | Fetch on-demand from API |
| Vector embeddings | Pre-computed, stored in FAISS | None — LLM reads raw history |
| Recommendation logic | KNN in vector space → LLM rank | Taste planner → TMDB discover → LLM picker |
| Similar Items search | KNN by TMDB ID or text | TMDB `/similar` API by TMDB ID |
| Discover data | FAISS index (static) | TMDB discover API (dynamic) |
| Will I Like? | Embedding similarity + explanation | LLM prediction (no embeddings) |
| Visual Explorer | t-SNE clusters of embeddings | (removed) |
| Admin UI features | FAISS rebuild, embed management | Sync status, cache clear only |
| MongoDB size | ~2 GB (metadata + history) | < 100 MB (history only) |
| Cold start latency | 10–30s (FAISS download) | < 2s (no index load) |
| GCP cost | $50–100/month | $0–10/month (single VPS option) |
| Scaling | Stateful (FAISS cache per process) | Stateless (any process same result) |
